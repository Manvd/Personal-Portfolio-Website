I'm Manav Dawawala, a Data Engineer and Python backend developer based in India. I've spent the past 3+ years building systems that move, transform, and serve data reliably — the kind of infrastructure that quietly powers dashboards, reports, and decisions that teams rely on every day.

My work started with Python scripts and SQL queries, and grew into designing full ETL/ELT pipelines, distributed task managers, data warehouses, and REST APIs. I care deeply about writing code that is readable, testable, and easy to hand off — because the best systems are the ones that other engineers enjoy working with.

Two projects I'm especially proud of: a production ETL Pipeline Framework that processes millions of rows daily with zero missed SLAs, and a Scalable Task Manager built with FastAPI, Celery, and Redis that handles 50,000+ jobs per day without breaking a sweat.

The best pipeline is the one nobody notices — because it always runs, always delivers, and never lies about its data.

Outside work I'm constantly learning — currently exploring Apache Flink, advanced dbt patterns, and preparing for my AWS Data Engineer certification. The data engineering field evolves fast and staying curious is the only way to keep up.

I value clear communication, honest feedback, and sustainable engineering practices. I don't build systems that need constant babysitting — I build systems that tell you when something's wrong before you even notice.

What I believe

Engineering principles

01
Reliability over cleverness
A boring pipeline that runs every time beats a clever one that occasionally doesn't. I optimise for predictability, not elegance for its own sake.
02
Observability first
If a pipeline fails silently, it might as well not exist. Logging, alerting, and data quality checks are not optional — they're part of the feature.
03
Idempotency matters
Every pipeline I write can be re-run safely. Duplicate data is worse than missing data — and it's entirely preventable with the right design.
04
Document the contract
Before writing a line of code, I define the data contract — schema, freshness SLA, quality rules. Surprises at 2am are a documentation failure.
05
Test early, test often
Unit tests, integration tests, and data quality assertions aren't overhead — they're the only way to refactor confidently and move fast without breaking things.
06
Keep learning
The data ecosystem evolves fast. I stay current, experiment with new tools, and bring back only what genuinely solves problems — not what's trendy.
